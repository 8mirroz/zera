# Runtime Control Plane v2 Rollback Procedure

## Triggers for Rollback

Execute this rollback procedure if any of the following conditions are met during or after the migration to `zera_runtime_control_plane.v2.json`:

1.  **Critical Startup Failure:** The Zera/Hermes runtime fails to initialize the core memory loop even after circuit breaker retries.
2.  **MCP Paralysis:** Essential tools (filesystem, context7, obsidian) become unavailable or fail health checks continuously, causing the `safe_default` profile to break.
3.  **Command Unresponsiveness:** Zera commands fail to execute due to unresolvable routing rules in the new orchestration layer.
4.  **Severe Workflow Disruption:** Standard workflows (like `/neo` or `/genesis`) consistently fail at the preflight or quality gate stages.

## Rollback Steps

The v2 implementation relies on a non-destructive overlay. Rolling back is straightforward.

### Step 1: Deactivate the v2 Overlay

Modify the primary bootstrap script or runner configuration to ignore the `zera_runtime_control_plane.v2.json` file.

Alternatively, if the runtime automatically loads all JSON files in the `configs/runtime/` directory:

```bash
# Rename the overlay to disable it
mv /Users/user/zera/configs/runtime/zera_runtime_control_plane.v2.json /Users/user/zera/configs/runtime/zera_runtime_control_plane.v2.json.disabled
```

### Step 2: Re-target the Base Config

Ensure the system explicitly targets the legacy `v1` configuration:

```bash
# Verify the base config is intact
ls -la /Users/user/zera/configs/runtime/zera_runtime.v1.json
```
*(If the v1 config was somehow modified or moved, restore it from Git history or backups).*

### Step 3: Flush Volatile State

Clear any corrupted session memory or cached routing decisions that might have been generated by the v2 logic.

```bash
# Clear agent session memory caches if applicable
# (Path depends on specific Zera implementation, usually within .agent/ or .gemini/ dirs)
```

### Step 4: Restart Runtime

Restart the Zera/Hermes agent process. Monitor the startup logs to ensure it initializes using the flat v1 schema without the capability matrices or staged startup logic.

### Step 5: Post-Rollback Diagnostics

1. Run a basic memory retrieval test.
2. Run a standard filesystem write test.
3. Review `/Users/user/zera/logs/runtime/` to identify the root cause of the v2 failure (e.g., schema mismatch, timeout, secret resolution failure).
4. Do not attempt a re-migration until the identified failure class has been patched in a new `.v2.1.json` overlay.
